# PPA-Lite 学习方案：与 Agent Harness 互补的设计与验证成长路径

## 1. 目标

本方案面向“已学过 SV/UVM，但尚未系统学习 APB 总线、缺少复杂项目 UVM 实战”的学习者。目标不是让 Agent 替你完成 `/ppa-lab/`，而是把现有 Agent Harness 变成“助教 + 代码审查员 + 调试伙伴”，让你逐步掌握：

1. APB 3.0 两阶段传输、CSR、PSLVERR、PREADY 等协议要点
2. PPA-Lite 的模块级 RTL 设计：M1 APB/CSR、M2 SRAM、M3 packet processor
3. 从定向 SV TB 到集成级验证，再到 Lab4 回归与覆盖率闭环
4. 读 spec、写 testplan、建 reference model、定位 mismatch 的工程方法
5. 与 Agent 协作时保持“自己负责技术判断”的工作方式

## 2. 现有工作流如何定位

`/ppa-lab/doc/CLAUDE.md` 已规定 Agent 进入项目后按 `ppa-status.md → labX/doc/handoff.md → ppa-feature-matrix.md → ppa-agent-character.md → ppa-lab-prompt.md/ppa-lite-spec.md` 的顺序工作。`ppa-agent-character.md` 又把工作拆成 DUT、Review、VPlan、VDebug、Sign-off 等角色。

这套机制适合“推进项目”，但如果完全交给 Agent，你只能学会如何管理 Agent。因此建议采用“双轨制”：

| 轨道 | 负责人 | 目的 | 典型产出 |
| --- | --- | --- | --- |
| 学习轨 | 你 | 建立协议、RTL、验证方法的个人理解 | 手写笔记、波形截图、预期值推导、问题清单 |
| 工程轨 | Agent | 按 harness 规范推进文档、代码、审查、回归 | RTL/TB、testplan、handoff、status、feature-matrix |

关键原则：**Agent 可以写代码，但不能替你做判断；Agent 可以解释结论，但你必须能用 spec、波形和测试结果复述为什么。**

## 3. 每次使用 Agent 前后的固定动作

### 3.1 Agent 前：先做“人类预习”

每次启动一个 Agent 任务前，先完成以下 5 件事：

1. 读本 lab 的 `design-prompt.md` / `testplan.md` / `acceptance.md`
2. 从 `ppa-feature-matrix.md` 中圈出本次相关 feature ID
3. 查 `ppa-lite-spec.md` 对应章节，只读相关小节，不通读全文
4. 手写 3 个问题：协议问题、RTL 问题、验证问题各 1 个
5. 先写出你自己的预期：这个 feature 应该如何实现、如何观察、如何判定 PASS

### 3.2 Agent 中：把 Agent 当“苏格拉底式助教”

向 Agent 提问时优先使用这类指令：

- “不要直接改代码，先指出我这段理解哪里可能错”
- “请按 spec 章节逐项挑战我的 testplan”
- “请给我 3 个最小反例，证明这个 checker 不充分”
- “请解释这个波形中 APB SETUP/ACCESS 每一拍发生了什么”
- “请把失败归因分成 RTL bug、TB bug、spec 歧义、环境问题”

尽量少用这类指令：

- “帮我把这个 lab 做完”
- “直接写完整 UVM 环境”
- “所有失败都帮我修掉”
- “覆盖率不够你直接补”

### 3.3 Agent 后：做“复盘验收”

Agent 完成后，不要只看 diff。至少完成：

1. 对照 `ppa-feature-matrix.md`，确认每个 #DONE/#VERIFIED 是否有 testcase 支撑
2. 从新增/修改的 TB 中挑 1 个 testcase，手算输入与期望输出
3. 打开 1 段关键波形，标出 SETUP、ACCESS、start、busy、done、irq 或 SRAM rd_data 的时序
4. 用自己的话补充 `log.md` 中没有写清楚的理解点
5. 整理“我下次不需要 Agent 也能做的部分”

## 4. 分阶段学习路线

### 4.1 Lab1：APB 从接口 + SRAM

当前状态：Lab1 已关闭，F1-01~F1-15 已验证。你应把它当成 APB 入门的复盘材料。

学习重点：

1. APB 两阶段传输：PSEL/PENABLE/PWRITE/PADDR/PWDATA/PRDATA/PSLVERR/PREADY
2. CSR 属性：RW、RO、W1P、RW1C
3. 地址译码与错误响应：CSR、PKT_MEM、reserved address
4. SRAM 同步读写与 APB 访问之间的时序差异

建议练习：

| 练习 | 要求 | 不让 Agent 做什么 |
| --- | --- | --- |
| 手画 APB 写时序 | 画出 SETUP、ACCESS 两拍，标出写入生效点 | 不让 Agent 直接给答案，先画后请它审查 |
| 手算 CSR 默认值 | 从 spec §5.2 推导 reset 后读值 | 不让 Agent 总结表格 |
| 设计 1 个非法地址用例 | 写出 PADDR、期望 PSLVERR、是否有副作用 | 不让 Agent 直接写 testcase |
| 复盘 W1P/RW1C | 解释 start 与 IRQ_STA 为什么不能按普通 RW 处理 | 不让 Agent 只给概念定义 |

通过标准：你能不用看 RTL，解释一次 APB 写 CTRL.start 触发处理的完整路径。

### 4.2 Lab2：Packet Processor FSM

当前状态：Lab2 已关闭，F2-01~F2-15 已验证。你应重点学习“模块级算法核 + FSM + reference model”。

学习重点：

1. IDLE → PROCESS → DONE 三态 FSM
2. SRAM 同步读延迟对 M3 `mem_rd_en_o` / `mem_rd_addr_o` 的影响
3. 包头解析：pkt_len、pkt_type、flags、hdr_chk
4. length/type/check 三类错误并行成立
5. payload sum/xor 的逐字节 reference model

建议练习：

| 练习 | 要求 | 不让 Agent 做什么 |
| --- | --- | --- |
| 画 FSM 状态转移 | 标出 start、done、busy、结果保持条件 | 不让 Agent 代画，先请它找漏洞 |
| 手算 8B 合法包 | 计算 header checksum、payload sum、payload xor | 不让 Agent 直接算最终值 |
| 构造三错误并发包 | 同时触发 length/type/chk error | 不让 Agent 直接生成 packet |
| 解释同步 SRAM 延迟 | 说明为什么要区分 issue_idx / consume_idx | 不让 Agent 只贴代码解释 |

通过标准：你能从一个 32-bit word 的 packet header 推导所有错误标志和处理结果。

### 4.3 Lab3：顶层集成与端到端验证

当前状态：Lab3 验证完成，待用户运行 `make comp && make run` 并由 Sign-off Agent 验收。

学习重点：

1. M1/M2/M3 在 `ppa_top` 的连线职责
2. APB 写 packet → 配置 CSR → start → 轮询 STATUS.done → 读 RES/ERR 的完整闭环
3. 端到端 testcase 与模块级 testcase 的区别
4. M2 单读端口在 M1 读回与 M3 处理之间的仲裁限制
5. 中断路径：IRQ_EN、IRQ_STA、irq_o、RW1C 清除

建议练习：

| 练习 | 要求 | 不让 Agent 做什么 |
| --- | --- | --- |
| 画端到端数据流 | 标出 APB、CSR、SRAM、M3、结果回传路径 | 不让 Agent 直接生成框图 |
| 复盘 TC1~TC11 | 每个 testcase 写一句“它证明了什么” | 不让 Agent 替你总结 |
| 亲自跑 Lab3 | 在本地 Questa 环境运行 make comp/run，记录失败/通过日志 | 不让 Agent 只根据代码声称通过 |
| 做一次 Sign-off 演练 | 对照 `acceptance.md` 自己判 PASS/FAIL | 不让 Agent 先给验收结论 |

通过标准：你能解释一个合法包从 APB 写入到 irq_o 拉高的每个可观测信号。

### 4.4 Lab4：回归测试与覆盖率闭环

当前状态：Lab4 尚未启动，F4-01~F4-06 为 #TODO。这是你最适合深度参与的阶段。

学习重点：

1. smoke / regress / cov 的目标区别
2. testcase 清单、seed、日志、result_summary 的组织方法
3. line/branch/condition/FSM/toggle 覆盖率的含义
4. 覆盖率缺口分析：是设计不可达、测试缺失，还是过滤规则问题
5. 从定向 SV TB 过渡到更系统的 UVM 组件化验证

建议采用“你先设计、Agent 后实现/审查”的顺序：

| 步骤 | 你先做 | Agent 后做 |
| --- | --- | --- |
| F4-01 smoke | 定义最小必跑用例集合 | 检查 Makefile 入口和日志输出 |
| F4-02 regress | 定义全量 testcase 列表和 PASS 判据 | 实现批量运行与汇总 |
| F4-03 cov | 说明要打开哪些覆盖率类型 | 补全工具参数 |
| F4-04 testplan | 先写覆盖矩阵草稿 | 挑战遗漏与重复项 |
| F4-05 覆盖率达标 | 先分析覆盖率报告 | 建议补测或过滤 |
| F4-06 summary | 先定义人类想看的摘要格式 | 自动生成 result_summary |

通过标准：你能看懂一次 regress/cov 结果，并判断下一步应补 testcase、修 RTL、修 TB，还是登记风险。

## 5. 推荐的个人学习产物

建议在你自己的笔记系统中维护以下内容，不一定提交到仓库：

1. **APB 时序卡片**：读/写各一张，包含 SETUP/ACCESS、PREADY、PSLVERR
2. **CSR 属性卡片**：RW、RO、W1P、RW1C 的读写副作用
3. **PPA-Lite 数据流图**：M1/M2/M3/Top 的信号连接
4. **Testcase 设计表**：每个 TC 的刺激、观测点、期望、覆盖 feature
5. **Bug 归因记录**：每次失败按 RTL/TB/spec/env 分类
6. **波形截图集**：每个 lab 至少保留 1 张关键波形并写解释

## 6. 与 Agent 协作的任务模板

### 6.1 学习型提问模板

```text
我正在学习 <lab/feature>，我自己的理解如下：
1. ...
2. ...
3. ...

请你只做审查，不要改代码：
- 哪些理解与 spec 不一致？
- 哪些信号/时序我遗漏了？
- 如果我要写 testcase，最小刺激和观测点是什么？
```

### 6.2 实现型任务模板

```text
请作为 <DUT/VPlan/VDebug/Review/Sign-off> Agent 处理 <feature ID>。
约束：
1. 先说明将修改哪些文件
2. 不要修改 spec
3. 每个 testcase 必须说明覆盖的 feature ID
4. 完成后更新 status/feature-matrix/handoff/log
5. 给出我需要亲自复盘的 3 个问题
```

### 6.3 Debug 型任务模板

```text
以下是我本地运行的失败信息：
- 命令：
- seed：
- testcase：
- 失败日志：
- 我观察到的波形：

请先归因，不要直接修：
1. 可能是 RTL/TB/spec/env 哪一类？
2. 最小复现路径是什么？
3. 需要查看哪些信号？
4. 如果要改，最小改动是什么？
```

## 7. 自我考核清单

每完成一个 lab，用下面问题判断是否真正学到：

### APB / CSR

- 我能否解释 APB SETUP 和 ACCESS 的区别？
- 我能否指出写操作在哪一拍生效？
- 我能否解释 PREADY 固定为 1 对 TB driver 的影响？
- 我能否解释 RO 写、reserved address、busy 写 PKT_MEM 的 PSLVERR 差别？
- 我能否解释 W1P 和 RW1C 的副作用？

### RTL 设计

- 我能否从 spec 端口表画出模块连接？
- 我能否解释每个 always_ff / always_comb 的职责？
- 我能否说明哪些寄存器保存状态，哪些信号只是组合译码？
- 我能否发现“把逻辑堆到 ppa_top”为什么不合适？

### 验证

- 我能否为每个 feature 写出至少一个 testcase？
- 我能否区分 driver、monitor、checker、scoreboard 的职责？
- 我能否手算 reference model 的期望值？
- 我能否从 mismatch 日志定位到可能的信号路径？
- 我能否判断覆盖率缺口是否真的需要补测？

### Agent 协作

- 我是否先写了自己的理解，再让 Agent 审查？
- 我是否能解释 Agent 生成的每个关键改动？
- 我是否能指出 Agent 方案的风险或遗漏？
- 我是否亲自跑过关键命令并看过日志/波形？

## 8. 建议的下一步

1. 先复盘 Lab1：手画 APB 写/读时序，并用 `ppa_apb_slave_if.sv` 对照

3. 亲自执行 Lab3 的 `make comp && make run`，再让 Sign-off Agent 验收
4. Lab4 开始前，先由你写一版 smoke/regress/cov 的 testplan 草稿，再交给 VPlan Agent 审查
5. 每次 Agent 交付后，至少回答本方案第 7 节中的 5 个问题，确认自己是在学习设计与验证，而不是只在审核 Agent 产物
