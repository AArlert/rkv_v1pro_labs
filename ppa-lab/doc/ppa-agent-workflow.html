2System Architecture

The PPA-Lite system consists of three core modules integrated into a top-level wrapper. Data flows from the APB bus through CSR/PKT_MEM into the SRAM, then processed by the FSM core.

3Agent Roles

Five specialized Agents collaborate through a handoff protocol. Each Agent has defined input contracts, trigger conditions, output requirements, and termination criteria.

DUT

DUT Agent

Write, verify, and fix RTL code.

  • Trigger: New lab starts or VDebug reports RTL bug
  • Input: design-prompt.md ready, feature-matrix rows #TODO/#WIP
  • Output: RTL passes make comp, feature-matrix updated to #DONE
  • Exit: All target rows #DONE + compile clean
VPLAN

Verification Plan Agent

Convert spec into testcase matrix and implement TB.

  • Trigger: DUT Agent completes current lab RTL
  • Input: Compiled RTL + design-prompt + spec
  • Output: testplan.md complete, TB code passes make run
  • Exit: All P0 testcases PASS, TB status column updated
VDEBUG

Verification Debug Agent

Analyze simulation failures, locate root cause.

  • Trigger: TB reports FAIL / ERROR / mismatch
  • Input: Failing sim log, reproducible make run
  • Output: Root cause located (file:line), fix recommendation in handoff
  • Exit: Root cause documented, or #BLOCKED in risk-register
INTEGRATION

Integration Agent

Maintain Makefile, directory layout, regression entry points.

  • Trigger: New lab setup or Lab4 regression build
  • Input: All sub-module RTL #DONE for current lab
  • Output: make comp/run/clean working; Lab4 adds smoke/regress/cov
  • Exit: Targets reproducible in clean environment
REVIEW

Review / Sign-off Agent

Structured review at each lab completion.

  • Trigger: Lab enters acceptance phase
  • Input: acceptance.md defined, DUT + Verification claim complete
  • Output: Each criterion judged PASS/FAIL in acceptance.md
  • Exit: All mandatory items PASS, or FAIL items logged to risk-register

5Session Lifecycle

Every Agent session follows a standardized 7-step lifecycle, ensuring consistent onboarding, execution, and handoff.

Onboarding (Entry)

1. Read ppa-status.md
Current progress, blockers, next steps (≤30s)
2. Read handoff.md
What the previous Agent left behind (≤2min)
3. Read ppa-feature-matrix.md
Which features are done, which are pending (≤2min)
4. Select scope
Target all feature-matrix rows for the current lab

Wrap-up (Exit)

5. Self-acceptance
Run acceptance.md criteria or make run
6. Write handoff
Append handoff block to handoff.md
7. Update status
Refresh ppa-status.md, ppa-feature-matrix.md, log.md

If time/context runs out mid-work: write handoff + status, then exit gracefully.

Each lab progresses through 5 phases. The phases form a cycle that can iterate until all acceptance criteria pass.

7Document Ecosystem — How Files Drive AI Agents

Documents are not passive records — they are the operating system of the Agent workflow. Every document has a specific role in the Agent's decision loop: what to read on entry, what to consult during work, and what to update on exit. The diagram below shows how information flows between them.

Governance Layer — Read-only for Agents, defines "how to think"

These files are the Agent's "operating instructions." They are set by the user and cannot be modified by Agents (changes require RFC via risk-register). Together they answer: What am I? What are the rules? What does the design look like?

CLAUDE.md Read-only
Agent Entry Point — the "front door" of every session
What it does for the Agent: This is the first file any Agent reads upon entering the project. It prescribes the exact onboarding sequence (what to read, in what order, with time budgets) and the wrap-up checklist (what to update before exiting). Without it, the Agent wouldn't know where to start.
Key mechanism: §1 defines a prioritized reading list (status → handoff → feature-matrix → character → prompt → spec). §2 defines mandatory exit actions. §3–4 set coding standards and conflict resolution rules.
Analogy: Like an employee handbook — read on day one, referenced when unsure about process.
ppa-lite-spec.md Read-only Priority #1
Technical Specification — the single source of truth
What it does for the Agent: When the Agent needs to know what to implement or verify, this is the authoritative answer. It defines APB protocol timing, CSR register layouts, FSM behavior, error codes, and more. All other documents defer to it when conflicts arise.
How the Agent uses it: The Agent does NOT read the entire spec upfront. Instead, it uses Spec § references from ppa-feature-matrix.md to jump to exactly the sections needed for the current task. For example, when implementing F1-03 (CSR address mapping), the Agent reads §3.2 of the spec.
Guard rail: No Agent may modify this file. If a spec error is found, the Agent logs it in ppa-risk-register.md as an RFC.
ppa-agent-character.md Read-only
Collaboration Protocol — defines "who does what" and "how to hand off"
What it does for the Agent: Tells each Agent its role boundaries: input contracts (what must be ready before I start), trigger conditions (what event activates me), output requirements (what I must produce), and termination criteria (when I'm done). Also defines the handoff protocol template — the structured format for session-to-session knowledge transfer.
Critical for multi-Agent coordination: Prevents role overlap (e.g., DUT Agent doesn't write testcases). Ensures each Agent knows which documents to read and update. The §8 table maps spec chapters to responsible Agents, so an Agent knows which parts of the spec are its concern.
ppa-lab-prompt.md Read-only
Project Blueprint — naming conventions, directory layout, coding constraints
What it does for the Agent: Provides the structural rules the Agent must follow: file naming (ppa_*.sv), directory organization (labX/rtl/, labX/svtb/), Makefile targets, and design/verification constraints. Also contains a quick-reference index of spec hard constraints (§0.4) so the Agent doesn't have to search the full spec for common rules.
Lab roadmap: §3 defines what each lab must accomplish and points to the corresponding acceptance.md. This is how the Agent understands the big picture of the project progression.
Shared State — Read on entry, write on exit, drives Agent decisions

These are the "live" documents that change every session. They form a feedback loop: each Agent reads the previous state, does work, and writes back the new state for the next Agent. This is how context survives across sessions without direct Agent-to-Agent communication.

ppa-status.md Read + Write
Progress Dashboard — the "30-second situational awareness" file
On entry (first file read, ≤30s): Tells the Agent the current project phase, what's been completed, what's blocked, and what the recommended next step is. This is a decision accelerator — the Agent immediately knows whether to start Lab3, debug a failure, or wait for a dependency.
On exit (mandatory update): The Agent refreshes the milestones table, in-progress items, and blocker list to reflect the session's work. The "next steps" section is especially important — it's a direct instruction to the next Agent.
Why this matters: Without ppa-status.md, every new Agent session would need to re-derive the project state from git history and code inspection. This file makes onboarding instant.
ppa-feature-matrix.md Read + Write
Task Board + Traceability — the Agent's "to-do list" and progress tracker
Core mechanism: Each row is an atomic feature with a unique ID (F1-01, F2-05, etc.), a Spec § link, implementation status, TB status, and associated testcase name. The Agent targets all rows in the current lab per session — since PPA-Lite's design is compact, each lab can be completed in a single session.
On entry: The Agent scans for #TODO and #WIP rows to identify available work. The Spec § column tells it exactly which spec sections to read. The Testcase column tells it which test to run for verification.
On exit: The Agent updates status columns (#TODO → #WIP → #DONE → #VERIFIED) to reflect work completed.
Traceability: This file replaces a separate traceability matrix. Every feature links back to the spec and forward to the test, so nothing falls through the cracks.
ppa-risk-register.md Read + Write
Escalation Path — where uncertainty gets captured, not ignored
When the Agent writes to it: Whenever the Agent encounters a spec ambiguity, makes an assumption, finds a potential bug in the spec, or cannot complete a task due to a blocker. Instead of guessing or stopping, it records the issue here with severity and continues with a documented assumption.
When the Agent reads it: On entry, to check if there are known risks that affect the current task. Before implementing a feature, to see if someone else already flagged an ambiguity in the relevant spec section.
Escalation mechanism: Modifying governance files (spec, character, prompt) requires an RFC logged here first. This prevents unauthorized changes while still allowing issues to be raised.
Per-Lab Working Docs — Lab-scoped context for focused execution

Each lab has its own set of working documents in labX/doc/. These provide lab-specific context that the project-level docs don't capture: detailed design decisions, test matrices, and inter-session communication.

design-prompt.md DUT Agent writes
Lab Design Guide — translates spec requirements into an implementation plan
Created by: DUT Agent, before writing any RTL. It bridges the gap between the abstract spec and the concrete implementation: which signals to define, how to structure the FSM, what edge cases to handle.
Used by: VPlan Agent reads it to understand design intent and write targeted tests. VDebug Agent reads it to understand expected behavior when diagnosing failures.
testplan.md VPlan Agent writes
Test Matrix — maps features to testcases with coverage goals
Created by: VPlan Agent, after RTL compiles. Each entry specifies: input stimulus summary, expected output, coverage target, and priority (P0/P1/P2).
Guard rail: No TB code may be written without a testplan first. This prevents ad-hoc testing and ensures systematic coverage. The testcase names here must match those in ppa-feature-matrix.md.
handoff.md All Agents append
Session Memory — how knowledge survives across Agent sessions
The most critical per-lab doc for continuity. Each Agent appends a structured block on exit: what was done (≤5 items), what remains, pitfalls encountered, minimal verification command, and recommended next actions. The next Agent reads this first to avoid repeating work or hitting the same traps.
Format: Strictly templated (From → To, date, lab) so entries are scannable. This is the Agent equivalent of a shift-change log.
acceptance.md Review Agent judges
Quality Gate — machine-checkable criteria that must all PASS to complete a lab
Purpose: Eliminates subjective "good enough" judgments. Each criterion is a concrete, verifiable check (e.g., "make comp exits with 0 errors," "tc_apb_basic_rw reports PASS"). The Review Agent executes each criterion and records PASS/FAIL.
On failure: FAIL items are logged to ppa-risk-register.md and trigger a new iteration cycle. The lab cannot be signed off until all mandatory criteria pass.
log.md All Agents write
Lab Journal — detailed record for human review
Key distinction: Unlike handoff.md (written for the next Agent), log.md is written for human readers who want to understand design decisions, challenges, and rationale. Agents write to it but typically only read the top Status summary (≤20 lines).
Structure: Organized by phase (Design → Review → Verification → Acceptance → Iteration), mirroring the lab workflow phases. Each phase records what was decided and why.

Priority Order (conflict resolution)

When documents contradict each other, higher-priority documents win:

ppa-lite-spec.md > ppa-agent-character.md > ppa-lab-prompt.md > CLAUDE.md > design-prompt.md

8Guard Rails

All Agents operate under strict boundaries to prevent accidental damage and ensure reproducibility.

Must Do

  • Follow onboarding & wrap-up checklists
  • Update feature-matrix, status, handoff every session
  • Record assumptions in risk-register
  • Complete all feature-matrix rows for the current lab per session
  • Graceful exit when context runs low

Must Not

  • Modify ppa-lite-spec.md (use RFC via risk-register)
  • Modify /lecture, /mcdt-lab directories
  • Modify .gitignore, README.md
  • Edit .sv files outside current task scope
  • Add features beyond what the task requires
