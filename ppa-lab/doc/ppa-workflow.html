PPA-Lite Agent Workflow

APB 数据包处理加速器 — AI Agent 协作总览

1项目概述

PPA-Lite 是一款基于 APB 3.0 总线接口的可编程数据包处理加速器。项目分为 4 个 Lab，每个 Lab 在前一个的基础上逐步推进，由专职 AI Agent 遵循结构化协作协议进行开发与验证。

2系统架构

PPA-Lite 系统由三个核心模块集成到顶层封装中。数据从 APB 总线经由 CSR/PKT_MEM 流入 SRAM，再由 FSM 核心处理。

3Agent 角色

五个专职 Agent 通过 Handoff 协议协同工作。每个 Agent 有明确的输入契约、触发条件、输出要求和终止判据。

DUT

DUT Agent

编写、验证并修复 RTL 代码。

  • 触发：新 Lab 开始，或 VDebug 上报 RTL 缺陷
  • 输入：design-prompt.md 就绪，feature-matrix 中存在 #TODO/#WIP 行
  • 输出：RTL 通过 make comp，feature-matrix 更新为 #DONE
  • 退出：所有目标行 #DONE 且编译无报错
VPLAN

Verification Plan Agent

将 spec 转化为测试用例矩阵并实现 TB。

  • 触发：DUT Agent 完成当前 Lab 的 RTL
  • 输入：已编译的 RTL + design-prompt + spec
  • 输出：testplan.md 完整，TB 代码通过 make run
  • 退出：所有 P0 测试用例 PASS，TB 状态列已更新
VDEBUG

Verification Debug Agent

分析仿真失败，定位根本原因。

  • 触发：TB 上报 FAIL / ERROR / mismatch
  • 输入：失败的仿真日志，可复现的 make run
  • 输出：定位到根本原因（file:line），修复建议写入 handoff
  • 退出：根本原因已记录，或在 risk-register 标注 #BLOCKED
INTEGRATION

Integration Agent

维护 Makefile、目录结构和回归测试入口。

  • 触发：新 Lab 环境搭建或 Lab 4 回归构建
  • 输入：当前 Lab 所有子模块 RTL 已 #DONE
  • 输出：make comp/run/clean 正常运行；Lab 4 新增 smoke/regress/cov
  • 退出：在干净环境中各目标可复现
REVIEW

Review / Sign-off Agent

在每个 Lab 完成时进行结构化评审。

  • 触发：Lab 进入验收阶段
  • 输入：acceptance.md 已定义，DUT 与验证声称完成
  • 输出：acceptance.md 中每条判据已评定 PASS/FAIL
  • 退出：所有必选项 PASS，或将 FAIL 项记录到 risk-register

5会话生命周期

每次 Agent 会话遵循标准化的七步生命周期，确保一致的入场、执行和交接。

入场流程（进入时）

1. 读取 ppa-status.md
当前进度、阻塞项、后续步骤（≤30 秒）
2. 读取 handoff.md
了解上一个 Agent 留下的内容（≤2 分钟）
3. 读取 ppa-feature-matrix.md
哪些功能已完成，哪些待完成（≤2 分钟）
4. 确定范围
以当前 Lab 的所有 feature-matrix 行为目标

收尾流程（退出时）

5. 自我验收
执行 acceptance.md 中的判据或 make run
6. 写入 handoff
向 handoff.md 追加交接记录
7. 更新状态
刷新 ppa-status.md、ppa-feature-matrix.md、log.md

若时间/上下文中途耗尽：先写入 handoff + status，再优雅退出。

6Lab 工作流阶段

每个 Lab 经历 5 个阶段。这些阶段构成一个循环，持续迭代直至所有验收判据通过。

7文档生态系统 — 文件如何驱动 AI Agent

文档不是被动的记录 — 它们是 Agent 工作流的操作系统。每份文档在 Agent 的决策循环中扮演特定角色：入场时读什么，工作时查什么，退出时更新什么。下图展示各文件之间的信息流向。

治理层 — Agent 只读，定义「如何思考」

这些文件是 Agent 的「操作说明书」。由用户设定，Agent 不得修改（变更需通过 risk-register 发起 RFC）。它们共同回答：我是谁？规则是什么？设计是什么样子？

CLAUDE.md Read-only
Agent 入口文件 — 每次会话的「前门」
对 Agent 的作用：这是每个 Agent 进入项目后首先读取的文件。它规定了精确的入场顺序（读什么、按什么顺序、时间预算是多少）以及收尾清单（退出前需更新什么）。没有它，Agent 将不知从何开始。
核心机制：§1 定义了优先读取列表（status → handoff → feature-matrix → character → prompt → spec）。§2 定义强制退出动作。§3–4 规定编码标准和冲突解决规则。
类比：如同员工手册 — 第一天阅读，流程不确定时查阅。
ppa-lite-spec.md Read-only Priority #1
技术规格书 — 唯一技术真相源
对 Agent 的作用：当 Agent 需要知道实现或验证什么时，此文件是权威答案。它定义了 APB 协议时序、CSR 寄存器布局、FSM 行为、错误码等内容。当其他文档出现矛盾时，均以本文件为准。
Agent 的使用方式：Agent 不会在开始时通读整个 spec。而是通过 ppa-feature-matrix.md 中的 Spec § 引用，直接跳转到当前任务所需的章节。例如，实现 F1-03（CSR 地址映射）时，Agent 读取 spec 的 §3.2。
约束：任何 Agent 均不得修改此文件。若发现 spec 错误，Agent 应在 ppa-risk-register.md 中记录 RFC。
ppa-agent-character.md Read-only
协作协议 — 定义「谁做什么」和「如何交接」
对 Agent 的作用：告知每个 Agent 的角色边界：输入契约（开始前必须就绪的内容）、触发条件（激活我的事件）、输出要求（我必须产出什么）、终止判据（我何时完成）。同时定义Handoff 协议模板 — 跨会话知识传递的结构化格式。
多 Agent 协调的关键：防止角色重叠（例如 DUT Agent 不编写测试用例）。确保每个 Agent 知道读取和更新哪些文档。§8 表格将 spec 章节映射到负责的 Agent，使 Agent 明确自己关注 spec 的哪个部分。
ppa-lab-prompt.md Read-only
项目蓝图 — 命名规范、目录结构、编码约束
对 Agent 的作用：提供 Agent 必须遵守的结构性规则：文件命名（ppa_*.sv）、目录组织（labX/rtl/、labX/svtb/）、Makefile 目标和设计/验证约束。还包含 spec 硬约束的快速索引（§0.4），使 Agent 无需搜索完整 spec 即可查阅常用规则。
Lab 路线图：§3 定义了每个 Lab 必须完成的内容，并指向对应的 acceptance.md。这是 Agent 理解项目整体推进路径的方式。
共享状态 — 入场读取，退出写入，驱动 Agent 决策

这些是每次会话都会变化的「活」文档。它们构成一个反馈环路：每个 Agent 读取前一次的状态，完成工作后将新状态写回，供下一个 Agent 使用。这就是上下文在无需 Agent 直接通信的情况下跨会话传承的机制。

ppa-status.md Read + Write
进度看板 — 「30秒态势感知」文件
入场时（首先读取，≤30秒）：告知 Agent 当前项目阶段、已完成内容、阻塞项和推荐的下一步。这是一个决策加速器 — Agent 立即知道是开始 Lab 3、调试失败，还是等待依赖项。
退出时（强制更新）：Agent 刷新里程碑表、进行中的条目和阻塞列表，以反映本次会话的工作成果。「后续步骤」部分尤为重要 — 它是给下一个 Agent 的直接指令。
重要性：没有 ppa-status.md，每次新 Agent 会话都需要从 git 历史和代码检查中重新推导项目状态。此文件使入场即时完成。
ppa-feature-matrix.md Read + Write
任务看板 + 可追溯性 — Agent 的「待办清单」与进度追踪器
核心机制：每行是一个原子功能，具有唯一 ID（F1-01、F2-05 等）、Spec § 链接、实现状态、TB 状态和关联测试用例名称。Agent 每次会话以当前 Lab 的所有行为目标 — 由于 PPA-Lite 设计紧凑，每个 Lab 可在单次会话中完成。
入场时：Agent 扫描 #TODO 和 #WIP 行以确定可用工作。Spec § 列告知需要读取哪些 spec 章节。Testcase 列告知用于验证的测试名称。
退出时：Agent 更新状态列（#TODO → #WIP → #DONE → #VERIFIED）以反映已完成的工作。
可追溯性：此文件取代了独立的追溯矩阵。每个功能向上链接到 spec，向下链接到测试，确保无遗漏。
ppa-risk-register.md Read + Write
上报路径 — 不确定性被记录而非忽视的地方
Agent 何时写入：每当 Agent 遇到 spec 歧义、做出假设、发现 spec 中的潜在缺陷，或因阻塞无法完成任务时。Agent 不是猜测或停下，而是在此记录问题（含严重程度）并以已记录的假设继续工作。
Agent 何时读取：入场时，检查是否存在影响当前任务的已知风险。实现功能前，查看是否有人已对相关 spec 章节标注了歧义。
上报机制：修改治理文件（spec、character、prompt）需先在此处记录 RFC。这在允许提出问题的同时防止未经授权的变更。
各 Lab 工作文档 — 聚焦执行的 Lab 级上下文

每个 Lab 在 labX/doc/ 中有自己的工作文档集。这些文档提供项目级文档未涵盖的Lab 特定上下文：详细的设计决策、测试矩阵和会话间通信。

design-prompt.md DUT Agent writes
Lab 设计指南 — 将 spec 要求转化为实现计划
创建者：DUT Agent，在编写任何 RTL 之前。它弥合了抽象 spec 与具体实现之间的差距：定义哪些信号、如何构建 FSM、需要处理哪些边界情况。
使用者：VPlan Agent 读取它以理解设计意图并编写针对性测试。VDebug Agent 读取它以在诊断失败时理解预期行为。
testplan.md VPlan Agent writes
测试矩阵 — 将功能映射到测试用例并设定覆盖率目标
创建者：VPlan Agent，在 RTL 编译完成后。每条目指定：输入激励摘要、预期输出、覆盖率目标和优先级（P0/P1/P2）。
约束：不得在 testplan 之前编写任何 TB 代码。这防止了随意测试，确保了系统性覆盖。此处的测试用例名称必须与 ppa-feature-matrix.md 中的一致。
handoff.md All Agents append
会话记忆 — 知识如何跨 Agent 会话传承
持续性最关键的 Lab 级文档。每个 Agent 退出时追加一个结构化记录：已完成内容（≤5 条）、剩余工作、遇到的注意事项、最小验证命令和推荐的后续动作。下一个 Agent 首先读取此文件，以避免重复工作或踩入相同陷阱。
格式：严格模板化（From → To、日期、Lab）使条目可快速扫读。这是 Agent 版的交班记录。
acceptance.md Review Agent judges
质量关卡 — 可机器核查的判据，全部 PASS 才能完成 Lab
目的：消除主观的「差不多够了」判断。每条判据都是具体、可验证的检查（例如「make comp 退出码为 0」、「tc_apb_basic_rw 报告 PASS」）。Review Agent 执行每条判据并记录 PASS/FAIL。
失败时：FAIL 条目记录到 ppa-risk-register.md 并触发新的迭代周期。所有必选判据全部通过前，Lab 不得签收。
log.md All Agents write
Lab 日志 — 供人工查阅的详细记录
关键区别：与 handoff.md（为下一个 Agent 撰写）不同，log.md 是为人工读者撰写的，供其了解设计决策、挑战和理由。Agent 会写入它，但通常只读取顶部的 Status 摘要（≤20 行）。
结构：按阶段组织（设计 → 评审 → 验证 → 验收 → 迭代），与 Lab 工作流阶段对应。每个阶段记录决策内容及理由。

优先级顺序（冲突解决）

当文档相互矛盾时，高优先级文档优先：

ppa-lite-spec.md > ppa-agent-character.md > ppa-lab-prompt.md > CLAUDE.md > design-prompt.md

8约束规则

所有 Agent 在严格约束下运行，以防意外损坏并确保可复现性。

必须做

  • 遵守入场和收尾清单
  • 每次会话更新 feature-matrix、status、handoff
  • 在 risk-register 中记录假设
  • 每次会话完成当前 Lab 的所有 feature-matrix 行
  • 上下文不足时优雅退出

禁止做

  • 修改 ppa-lite-spec.md（须通过 risk-register 发起 RFC）
  • 修改 /lecture、/mcdt-lab 目录
  • 修改 .gitignore、README.md
  • 编辑当前任务范围之外的 .sv 文件
  • 添加任务要求之外的功能
