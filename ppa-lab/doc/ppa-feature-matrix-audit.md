## 2 缺失功能点（Spec 有定义但 Feature Matrix 未覆盖）

| # | 缺失功能 | Spec 出处 | 建议归属 | 严重程度 |

| M-1 | PKT_LEN_EXP 与 pkt_len 不符时触发 length_error | §9.1: "pkt_len ≠ exp_pkt_len（若 PKT_LEN_EXP 已配置）"；§10.3 B-4 | Lab2 或 Lab3 | **中** — spec 明确定义了此触发条件，feature matrix F2-05 仅覆盖了范围检查 [4,32]，遗漏了 exp_pkt_len 一致性检查 |

| M-3 | STATUS.error 聚合逻辑 | §5.2: "STATUS[2] error = ERR_FLAG 各位的或"；§9.2 | Lab2 或 Lab3 | **低** — 属于 STATUS 读回通路的一部分，可视为 F3-04 隐含覆盖 |

## 3 可能的多余/偏差项

| ID | 问题 | 分析 |
|----|------|------|
| F1-14 | "M2 复位清零" — spec 未明确要求 SRAM 复位清零 | spec §2.2 仅描述 M2 为"双端口同步 SRAM"，未要求复位后内容全零。**不影响功能正确性**，作为实现选择可保留，建议备注中标注"实现约定，非 spec 强制" |
| F2-02 | "start 仅在 enable=1 且 busy=0 接受" — 归属 Lab2 但实际由 M1 实现 | spec §5.2 CTRL.start 的 enable/busy 门控逻辑在 M1 内完成（M3 端口表无 enable_i）。此功能验证归属 Lab1（F1-08 已部分覆盖），放在 Lab2 会导致模块归属混乱。**建议**：保留在 Lab2 作为集成验证视角，备注说明"M1 门控，Lab2 验证 M3 端接受行为" |
